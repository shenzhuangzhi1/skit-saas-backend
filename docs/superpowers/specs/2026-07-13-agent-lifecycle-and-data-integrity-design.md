# 代理商生命周期与数据完整性重构设计

日期：2026-07-13

状态：已批准
范围：`skit-saas-backend`、`skit-saas-frontend`

## 目标

将“用户管理”中的代理商管理收敛为短剧 SaaS 唯一入口，使代理商、租户、管理员手机号、广告账号、邀请体系、分成数据与 App 发布档案保持一致，并让数据库升级可以随后台版本自动、可追踪地完成。

## 不变量

1. 只有平台 `super_admin` 可以创建、修改、归档、恢复代理商及管理全局租户套餐。
2. 每个代理商对应一个 `system_tenant`，内部仍使用数字 `tenant_id`；代理商管理员手机号是唯一业务身份，同时写入管理员的 `username` 和 `mobile`。
3. App 普通用户存储于 `skit_member`，手机号只要求租户内唯一；不同代理商租户允许存在相同普通用户手机号。
4. 新代理商由后台自动绑定编码为 `SKIT_AGENT_STANDARD` 的“代理商标准套餐”，客户端不得提交或修改套餐。
5. 代理商不物理删除，只能归档和恢复。归档保留会员、邀请关系、广告收益事件、分成账本、广告配置和 App 发布历史。
6. 分成账本和已发生收益事件属于不可变审计数据，不提供删除接口。
7. 广告密钥永不通过查询接口返回，空值不覆盖旧密钥，显式清除必须使用专用操作。

## 创建与编辑体验

创建表单保留以下核心字段：

- 代理商名称
- 登录手机号
- 初始密码
- 有效期
- 创建后是否立即启用

删除以下框架字段：联系人、独立管理员账号、租户套餐、后台账号额度。服务端自动设置联系人为代理商名称、`username = mobile`、后台账号额度为 `1`。

穿山甲和 Taku 放在可选配置区。每个平台有显式启用开关；未启用时可以留空，启用时必须通过服务端完整性校验。Taku S2S Secret 保持可选。编辑表单不混入密码重置，手机号换绑和密码重置使用独立对话框与独立接口。

有效期必须晚于当前时间。前端按所选日期的当天 `23:59:59` 提交，后端再次校验，避免创建后立即过期。

## 生命周期与 CRUD

代理商端点提供：

- 数据库分页列表、详情、创建、编辑
- 归档、恢复
- 换绑管理员手机号
- 重置管理员密码
- 轮换根邀请码
- 广告平台启用、停用、更新和显式清除凭证

归档在同一事务中完成：

1. 标记 `skit_agent.archived_time`；
2. 禁用 `system_tenant`；
3. 禁用绑定管理员并撤销登录令牌；
4. 所有 App 上下文、邀请注册、广告收益回调和 App 更新入口统一校验租户有效性。

恢复执行反向状态更新，但不会自动重新启用不完整的广告平台配置。

普通会员不允许后台手工创建或物理删除，来源只能是手机号注册与邀请码。后台提供详情、启用/停用与密码重置。已归档代理商的数据仍可由 `super_admin` 审计查询。

## 服务端模型

创建、基础编辑、手机号换绑、密码重置和生命周期操作使用不同 Request VO，避免一次普通编辑意外修改密码或身份。

`system_tenant.status` 是可用状态的唯一事实来源。`skit_agent` 不再被单独状态判断；保留旧列仅用于兼容迁移，并由代码停止读取，新增 `archived_time` 和 `archived_by` 表示归档事实。

代理商列表使用数据库分页查询代理商与租户核心信息，再对当前页批量补充管理员、套餐和广告配置，禁止全表加载后内存分页和逐代理商 N+1 查询。

## 数据库升级与约束

用带版本记录和 checksum 的 `skit_schema_migration` 替换无版本的启动补丁。每次后台镜像启动时只执行尚未应用的迁移；执行失败则停止启动，避免应用在半升级结构上运行。全新库的 bootstrap SQL 与迁移后的最终结构保持一致。

本次迁移包括：

- `system_tenant_package.code`，有效记录唯一；标准套餐编码为 `SKIT_AGENT_STANDARD`；
- `skit_agent.archived_time`、`archived_by`；
- `system_users` 有效用户名和有效手机号的数据库级全局唯一约束；
- 代理商、会员、分成方案、收益事件与账本所需的查询和幂等索引；
- 状态、层级、比例、金额与有效期的应用校验和可行的数据库约束；
- 存量代理商管理员身份修复：在无冲突时把管理员 `username/mobile` 同步为租户联系手机号；发现无效或重复手机号时迁移明确失败并报告冲突记录，不静默覆盖。

软删除唯一性通过“仅有效记录参与唯一约束”的生成列实现，避免 `(value, deleted)` 在多条历史删除记录上的冲突。

## 权限与密钥

全局租户套餐控制器增加 `super_admin` 与平台租户双重门禁。标准代理商套餐不授予系统用户、租户或套餐管理权限，代理商只能调用角色明确允许的短剧 SaaS 租户内接口。

广告字段继续使用加密 TypeHandler，查询只返回 `configured` 状态。加密主密钥必须通过部署环境变量或 Secret 注入；生产配置缺失时后台拒绝启动，不使用仓库默认密钥。

所有写接口补齐状态枚举、手机号、长度、有效期、广告平台字段组合和分成比例校验。跨租户操作继续以当前登录用户的原始租户身份判断，不信任 visit-tenant header。

## 兼容和发布

前端先兼容读取旧字段，但不再发送套餐、账号额度、联系人和独立用户名。后端发布后会自动升级数据库并同时支持新前端。存量代理商在迁移成功后自动使用手机号登录；平台 `super_admin` 不参与该迁移。

发布顺序为后端、前端；两个仓库分别通过 GitHub Actions 构建部署。发布后验证平台管理员创建/编辑/归档/恢复、代理商手机号登录与租户隔离、普通用户邀请码注册、广告配置不回显、归档后的所有入口拒绝访问。

## 验证标准

- 后端代理商、权限、迁移、身份、归档和密钥相关测试通过；
- 前端类型检查、单元测试和生产构建通过；
- 数据库迁移可重复启动且不会重复执行；
- `tenant_admin` 调用全局套餐和代理商管理接口返回无权限；
- 手机号换绑后旧手机号不能登录，新手机号只进入绑定租户；
- 归档后管理员、App 用户、邀请、收益回调和发布清单均被拒绝，恢复后按原数据继续运行；
- 分成和广告历史数据在归档、恢复过程中保持不变。
